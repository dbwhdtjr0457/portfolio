# 포트폴리오 사이트 구현 계획

본 문서는 [`spec.md`](/home/dbwhd/portfolio/spec.md)를 원천 데이터로, [`research.md`](/home/dbwhd/portfolio/research.md)를 방향성과 우선순위 기준으로 삼아 작성한 구현 직전 설계 문서다. 문서에 없는 사실은 추가하지 않았으며, 불명확한 항목은 `추가 확인 필요` 또는 `불확실`로 표기한다.

## 1. 사이트 목표

이 사이트의 목적은 실무 경험과 프로젝트를 한 번에 읽을 수 있는 `읽기 쉬운 자기소개형 포트폴리오`를 만드는 것이다.  
강한 퍼스널 브랜딩 랜딩페이지처럼 보이지 않되, 상단 소개와 대표 사례에서는 분명한 첫 인상을 남겨야 한다.  
전체 구조는 문서형에 가깝게 유지하되, 카카오페이 경험과 대표 프로젝트가 묻히지 않도록 시각적 위계를 둔다.  
무엇을 과하게 하지 않을지보다, `카카오페이 실무 경험`, `프로젝트별 성격 차이`, `프론트엔드로서의 역할 범위`를 분명히 보여주는 데 집중한다.

## 2. 페이지 전략

한 페이지 구조가 적절한 이유는 현재 공개할 콘텐츠가 `카카오페이 1개 + 프로젝트 3개 + 기술/활동/학력`으로 비교적 명확하게 정리되기 때문이다. 별도 상세 페이지를 만들기보다 한 흐름 안에서 읽게 하는 편이 이 포트폴리오의 성격과 맞다.

이력서형 구조와 포트폴리오형 요소를 섞는 이유는 다음과 같다.

- 이력서형 구조가 필요한 이유: 경력과 프로젝트를 빠르게 훑어볼 수 있어야 한다.
- 포트폴리오형 요소가 필요한 이유: Intro와 대표 사례에서는 단순 목록보다 한 단계 높은 시각적 위계가 있어야 기억에 남는다.
- 혼합 방식: 섹션 구조는 문서형으로 유지하고, Intro와 대표 사례 카드만 강조한다.

독자 흐름은 다음을 기준으로 설계한다.

- 채용 담당자: 상단 소개 확인 → 카카오페이 경험 확인 → 프로젝트 제목과 설명 훑기 → 학력/기술 확인
- 실무 프론트엔드 면접관: 상단 소개 확인 → 카카오페이 상세 읽기 → 대표 프로젝트 읽기 → 기술/활동 확인
- 협업 관점의 팀원: 경험과 프로젝트 확인 → 활동/협업 방식 확인 → 링크 확인

## 3. 정보구조(IA)

이 사이트의 기본 IA는 아래 순서를 따른다.

1. Intro
2. Summary
3. Experience
4. Selected Project
5. Other Projects
6. Skills
7. Activities
8. Education
9. Links

이 순서를 제안하는 이유는 다음과 같다.

- Intro에서 직무와 사이트 성격을 먼저 인지시킨다.
- Summary는 강점 슬로건이 아니라, 이 사이트를 어떻게 읽으면 되는지 안내하는 짧은 요약 역할을 맡는다.
- Experience에서 가장 중요한 실무 경험인 카카오페이를 먼저 보여준다.
- Projects는 대표 1개와 보조 2개로 구분해 평평한 나열을 피한다.
- Skills, Activities, Education, Links는 본문을 읽고 난 뒤 확인하는 보조 정보로 배치한다.

## 4. 각 섹션의 목적과 포함 콘텐츠

### Intro

- 목적: 이름, 직무, 짧은 소개로 첫 인상을 만든다.
- 포함 콘텐츠: 이름, 직무, 1~2문장 소개, 외부 링크, 하단 섹션으로 이동하는 CTA
- 위치 이유: 방문자가 가장 먼저 이 사이트의 성격을 파악해야 하기 때문이다.
- 톤: 담백하지만 밋밋하지 않게
- 제약: 문장 수는 최대 2문장, 링크는 2~4개 이내

### Summary

- 목적: 강점 나열이 아니라 이 사이트의 읽는 기준을 짧게 제공한다.
- 포함 콘텐츠: 경력 1개, 프로젝트 3개, 기술 범위 요약 정도의 짧은 항목
- 위치 이유: Experience에 들어가기 전에 전체 구성을 빠르게 이해시키기 위해서다.
- 톤: 설명형, 비과장형
- 제약: 3개 이내, 각 항목은 한 줄 수준

### Experience

- 목적: 가장 중요한 실무 경험인 카카오페이를 중심으로 신뢰도를 형성한다.
- 포함 콘텐츠: 회사명, 직무, 기간, 수행 축, 협업 맥락, 주요 작업 요약
- 위치 이유: 이 포트폴리오에서 가장 먼저 읽혀야 할 핵심 경력이다.
- 톤: 가장 단단하고 구체적이어야 한다.
- 제약: 본문은 길어질 수 있으나, 세부 작업은 4~6개 수준으로 정리한다.

### Selected Project

- 목적: Experience 다음으로 읽히는 대표 프로젝트를 통해 포트폴리오다운 인상을 남긴다.
- 포함 콘텐츠: 프로젝트명, 짧은 설명, 역할, 핵심 구현 포인트, 사용 기술, 링크
- 위치 이유: 실무 경험 다음에 프로젝트 설계력을 보여주는 대표 사례가 와야 균형이 맞는다.
- 톤: 문서형 구조 안에서 한 단계 강조
- 제약: 1개만 배치하고, 너무 긴 서술 대신 4~5개 포인트로 정리한다.

### Other Projects

- 목적: 기술 범위와 경험 폭을 보여주되, 본문 리듬을 지나치게 무겁게 만들지 않는다.
- 포함 콘텐츠: Sleep Now!, PONG 각 프로젝트의 짧은 설명, 역할, 핵심 포인트, 링크
- 위치 이유: 대표 프로젝트 이후에 보조 프로젝트를 짧게 읽게 해야 정보 흐름이 자연스럽다.
- 톤: 간결하고 비교 가능한 톤
- 제약: 프로젝트당 3~4개 포인트 이내

### Skills

- 목적: 경험을 읽고 난 뒤 기술 스택을 정리해 준다.
- 포함 콘텐츠: 프론트엔드, 환경/도구, 3D/실시간, 모바일 관련 기술군
- 위치 이유: 기술이 앞에 오면 일반적인 이력서처럼 보이기 쉽다.
- 톤: 정리형
- 제약: 숙련도 바, 점수, 퍼센트 표현은 사용하지 않는다.

### Activities

- 목적: 협업과 기록 방식을 보조적으로 보여준다.
- 포함 콘텐츠: 스크럼, 회의록, 개발일지, 멘토링, 문서화 자산
- 위치 이유: 핵심 경력과 프로젝트를 읽고 난 뒤 보완 정보로 보는 편이 적절하다.
- 톤: 차분한 보조 정보
- 제약: 활동 하나하나를 독립 사례로 키우지 않는다.

### Education

- 목적: 공식 스펙을 간결하게 제공한다.
- 포함 콘텐츠: 학력, 관련 이수, 자격증, 어학
- 위치 이유: 신뢰도 보완 정보이지만 첫 인상을 결정할 요소는 아니다.
- 톤: 가장 문서형에 가깝게
- 제약: 사실만 정리하고 해석 문구는 넣지 않는다.

### Links

- 목적: 외부 증빙과 추가 확인 경로를 제공한다.
- 포함 콘텐츠: 프로젝트 GitHub 링크, 추가 링크
- 위치 이유: 읽기를 마친 뒤 이동할 수 있게 하는 보조 영역이 적절하다.
- 톤: 기능적
- 제약: `spec.md`에 없는 링크는 `추가 확인 필요`로 남긴다.

## 5. Intro 섹션 설계

Intro는 강한 Hero 랜딩이 아니라, `짧은 자기소개 + 명확한 구조 안내`를 제공하는 상단 영역으로 설계한다.

배치 기준은 다음과 같다.

- 1행: 이름
- 2행: 직무
- 3행: 짧은 소개 문장 1~2문장
- 4행: 링크 버튼 또는 텍스트 링크
- 5행: Experience 또는 Selected Project로 이동하는 CTA

포함 요소는 아래 범위로 제한한다.

- 이름
- 직무: 프론트엔드 개발자
- 짧은 소개: A1을 압축한 소개 문장
- 링크: 프로젝트 GitHub 링크는 가능, 기타 링크는 `추가 확인 필요`
- CTA: `경력 보기`, `프로젝트 보기` 정도의 낮은 톤 버튼

밋밋하지 않게 만드는 장치는 다음 정도로 제한한다.

- 이름과 직무는 타이포 위계를 분명하게 둔다.
- 소개 문장은 본문보다 약간 큰 크기로 보여준다.
- 링크와 CTA는 작은 묶음으로 정리해 행동 지점을 만든다.
- 배경은 과한 그래픽 대신 여백과 구분선, 얕은 색 대비로 정리한다.

피해야 할 방식은 다음과 같다.

- 큰 슬로건 한 줄을 화면 대부분에 배치하는 Hero
- 과한 애니메이션, 카운터, 배지 나열
- 성과 수치 중심의 마케팅 카피

## 6. Summary 섹션 설계

Summary 섹션은 `필요하다`. 다만 강점 3개를 강하게 어필하는 섹션이 아니라, `이 사이트에 무엇이 들어 있는지`를 짧게 요약하는 구조가 적절하다.

권장 형태는 3개의 짧은 요약 항목이다.

1. 카카오페이 FE 인턴 경험
2. 3개의 프로젝트 경험
3. 사용 기술 범위

이 섹션의 역할은 독자가 본문을 읽기 전에 구조를 이해하도록 돕는 것이다. 따라서 "운영 리스크를 줄이는 프론트엔드" 같은 강한 해석보다, `실무 경험 / 프로젝트 / 기술 범위` 수준의 사실 기반 요약이 적절하다.

표현 방식은 다음을 따른다.

- 카드 3개 또는 한 줄 요약 3개
- 각 항목은 제목 + 보조 설명 1줄
- 장점 해석보다 범위와 맥락 설명 중심

## 7. Experience 섹션 설계

Experience 섹션은 카카오페이 하나만 다루는 구조로 설계한다. C1, C2, C3, C4를 독립 사례로 분리하지 않고, `카카오페이 FE 인턴` 아래에 주요 작업을 묶어 배치한다.

권장 구조는 다음과 같다.

- 상단 헤더: 회사명 / 역할 / 기간
- 짧은 설명: 수행 축 요약
- 본문: 주요 작업 리스트
- 하단 보조 정보: 협업 방식, 공개 범위 관련 주석

주요 작업 리스트는 아래 수준으로 정리한다.

1. 오퍼 고도화
2. 카카오페이포인트 및 운영 백로그
3. 건틀렛 UI 변경
4. Node 16→22 / yarn→pnpm 이관

각 작업에 포함할 정보는 다음과 같다.

- 무엇을 다룬 작업인지
- 내가 맡은 범위
- 결과 또는 의미
- 필요하면 `추가 확인 필요` 표기

표시 밀도는 Experience가 사이트 전체에서 가장 높아도 된다. 다만 한 작업마다 장문의 문단을 두기보다, `제목 + 2~4개 포인트` 구조가 적절하다. 실무 경험을 자세히 보여주되, 페이지가 이 섹션 하나로 끝나 보이지 않게 조절해야 한다.

## 8. Project 섹션 설계

Project 섹션은 `대표 프로젝트 1개 + 보조 프로젝트 2개` 구조가 적절하다. research.md의 기본 방향인 `프로젝트당 1개 항목`은 유지하되, 화면상 위계는 나눠서 준다.

### 대표 프로젝트

대표 프로젝트는 별숲을 우선 후보로 둔다.

- 이유: 3D UX, 카메라 이동, API 처리 규칙, React-Query 공통화, SPA 이동 설계 등 설명할 내용이 가장 풍부하다.
- 역할: Experience 이후 포트폴리오다운 인상을 남기는 프로젝트
- 방식: 상세형 카드 또는 넓은 섹션형

### 보조 프로젝트

보조 프로젝트는 Sleep Now!, PONG로 구성한다.

- Sleep Now!: React Native, 센서, 예측, 알림 중재 성격을 간결하게 정리
- PONG: 대시보드 UI, Three.js, WebSocket, 게임 로직 기여를 간결하게 정리
- 방식: 리스트형 또는 2열 카드형

밀도 기준은 다음과 같다.

- 대표 프로젝트: 제목, 한 줄 설명, 역할, 핵심 포인트 4~5개, 기술, 링크
- 보조 프로젝트: 제목, 한 줄 설명, 역할, 핵심 포인트 3개 내외, 링크

실무 경험과 프로젝트가 따로 놀지 않게 하려면 다음 원칙을 지킨다.

- Experience와 Project 모두 동일한 서술 프레임을 사용한다.
- 예: `무엇을 한 경험인지 → 맡은 범위 → 핵심 작업`
- 시각적 톤은 통일하되, Experience와 Selected Project만 한 단계 더 강조한다.

프로젝트 순서 자체는 `추가 확인 필요`다. 다만 현재 기준안은 `별숲 → Sleep Now! → PONG`이다.

## 9. Skills / Education / Activities / Links 섹션 설계

### Skills

- 기술은 카테고리형으로 정리한다.
- 예: `Frontend`, `Data/State`, `3D/Realtime`, `Mobile`
- 각 기술은 이름만 나열하거나 짧은 그룹 설명만 둔다.
- 숙련도 점수, 레이더 차트, 퍼센트는 사용하지 않는다.

### Education

- 학력, 관련 이수, 자격증, 어학을 한 섹션 안에 간결하게 넣는다.
- 학점은 `spec.md`에 있으므로 표시 가능하다.
- 해석 문구 없이 사실만 정리한다.

### Activities

- 활동은 별도 프로젝트처럼 키우지 않고, 협업/운영 방식 정리로 제한한다.
- 스프린트, 스크럼, 회의록, 개발일지, 멘토링, 문서화 자산 정도를 묶어 보여준다.
- 각 항목은 한 줄 요약이면 충분하다.

### Links

- 우선 포함 가능한 링크는 F 섹션의 GitHub 저장소 링크다.
- GitHub 프로필, 이력서 파일, 이메일, 블로그 링크는 `추가 확인 필요`.
- 링크 섹션은 단독 마지막 섹션 또는 Footer에 배치한다.

## 12. 디자인/UX 원칙

전체 톤은 `읽기 쉬운 문서형 UI`를 기본으로 한다. 시각적 강조는 Intro와 대표 사례에서만 제한적으로 사용한다.

### 기본 원칙

- 타이포 중심으로 위계를 만든다.
- 여백과 구분선으로 섹션을 나눈다.
- 카드 사용은 필요한 곳만 제한적으로 한다.
- 본문 폭은 너무 넓지 않게 유지한다.
- 모바일에서도 섹션 순서와 위계가 그대로 유지되게 한다.

### 타이포 기준

- 이름과 주요 섹션 제목은 분명한 크기 차이를 둔다.
- 본문은 읽기 쉬운 줄 길이와 행간을 우선한다.
- 과장된 디스플레이 서체보다 안정적인 서체가 적절하다.

### 여백과 구분 기준

- 섹션 간 간격은 충분히 둔다.
- Experience와 Projects는 내부 아이템 사이 간격도 명확히 둔다.
- 구분선은 얇고 절제되게 사용한다.

### 카드 사용 기준

- Intro: 카드가 아니라 열린 레이아웃이 적절하다.
- Experience: 카드보다는 섹션형 레이아웃이 적절하다.
- Selected Project: 카드형 강조 가능
- Other Projects: 간결한 카드형 가능

### 금지 사항

- 전체 화면을 덮는 Hero 스타일
- 과한 패럴랙스, 스크롤 인터랙션, 자동 슬라이드
- 기술 숙련도 퍼센트, 레이더 차트, 과도한 배지
- 화려한 배경 그래픽, 불필요한 3D 연출
- 모든 섹션을 동일한 카드형으로 반복하는 구성

## 15. 최종 요약

이 문서를 기준으로 구현되는 사이트는 한 페이지 구조의 자기소개형 포트폴리오다.  
상단에서는 이름, 직무, 짧은 소개로 첫 인상을 만들고, 이후에는 카카오페이 실무 경험을 가장 먼저 보여준다.  
프로젝트는 별숲을 대표 사례로 두고, Sleep Now!와 PONG를 보조 프로젝트로 배치하는 구성을 기본안으로 삼는다.  
전체 UI는 문서형에 가깝게 유지하되, Intro와 대표 프로젝트만 한 단계 더 강조한다.  
기술, 활동, 학력, 링크는 후반부에서 간결하게 정리해 본문 리듬을 해치지 않게 한다.  
즉, 이 사이트는 과한 브랜딩 랜딩도 아니고, 평평한 이력서도 아닌, 읽기 쉬우면서도 핵심 경험이 선명하게 남는 포트폴리오를 목표로 한다.
