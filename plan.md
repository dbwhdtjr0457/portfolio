# 포트폴리오 사이트 구현 계획

본 문서는 [`spec.md`](/home/dbwhd/portfolio/spec.md)를 원천 데이터로, [`research.md`](/home/dbwhd/portfolio/research.md)를 방향성과 우선순위 기준으로 삼아 작성한 구현 직전 설계 문서다. 이번 버전은 새로 상세화된 공식 스펙(A4-A5), PDF 증빙(F), GitHub 링크(G)를 반영한다. 문서에 없는 사실은 추가하지 않았으며, 불명확한 항목은 `추가 확인 필요` 또는 `불확실`로 표기한다.

## 1. 사이트 목표

이 사이트의 목적은 `카카오페이 실무 경험`, `프로젝트 3개`, `공식 이력과 증빙`을 한 흐름 안에서 읽을 수 있는 포트폴리오를 만드는 것이다.

핵심 목표는 다음과 같다.

- 첫 화면에서 프론트엔드 개발자 포트폴리오라는 점이 즉시 이해될 것
- 카카오페이 경험이 가장 먼저 신뢰도를 형성할 것
- 별숲, PONG, Sleep Now!가 서로 다른 기술 성격으로 읽힐 것
- 학력, 교육/수료, 자격증, 어학이 `정리된 공식 이력`으로 보일 것
- PDF 증빙과 GitHub 링크가 본문을 방해하지 않으면서 뒤에서 확인 가능할 것

즉, 이전보다 더 분명하게 `읽는 본문`과 `검증 자료`를 나누는 것이 이번 설계의 핵심이다.

## 2. 페이지 전략

한 페이지 구조가 여전히 가장 적절하다. 현재 공개할 핵심 콘텐츠는 `경력 1개 + 프로젝트 3개 + 활동/협업 + 공식 이력 + 증빙`으로 비교적 선형 흐름에 맞기 때문이다.

다만 이번 버전에서는 페이지 안의 역할을 다음 두 층으로 나눈다.

- 본문 층: Intro, Snapshot, Timeline, Experience, Projects, Activities, Education/Credentials
- 검증 층: Evidence, External Links

이렇게 나누는 이유는 다음과 같다.

- 포트폴리오는 먼저 읽혀야 한다.
- 증빙은 신뢰도를 높이지만, 본문 전개를 주도하면 안 된다.
- 새 PDF 자료가 많아졌기 때문에, 본문 카드와 증빙 링크를 한 섹션에 섞으면 정보 구조가 무거워진다.

독자 흐름은 아래를 기준으로 설계한다.

- 채용 담당자: Intro 확인 → 카카오페이 확인 → 프로젝트 훑기 → 공식 이력 확인 → 필요 시 증빙 열람
- 실무 면접관: Intro 확인 → 카카오페이 상세 읽기 → 프로젝트 상세 읽기 → 활동/협업 확인 → 필요 시 GitHub 확인
- 서류 검토 관점: 공식 스펙 확인 → PDF 증빙 확인

## 3. 정보구조(IA)

기본 IA는 아래 순서를 따른다.

1. Intro
2. Snapshot
3. Timeline
4. Experience
5. Projects
6. Activities
7. Education & Credentials
8. Evidence
9. External Links

이 순서를 제안하는 이유는 다음과 같다.

- Intro에서 이름, 직무, 사이트 성격을 즉시 이해시킨다.
- Snapshot은 이 사이트에 무엇이 들어 있는지 한 번에 요약한다.
- Timeline은 A5의 강점을 살려 연혁을 짧게 정리한다.
- Experience와 Projects에서 핵심 경쟁력을 먼저 보여준다.
- Activities는 일하는 방식을 뒤에서 보강한다.
- Education & Credentials는 이제 별도 섹션으로 충분한 밀도를 가진다.
- Evidence와 External Links는 마지막에 배치해, 읽기를 마친 뒤 추가 확인이 가능하게 한다.

## 4. 각 섹션의 목적과 포함 콘텐츠

### Intro

- 목적: 이름, 직무, 짧은 소개, 핵심 링크로 첫 인상을 만든다.
- 포함 콘텐츠: 이름, 직무, A1 압축 소개 1~2문장, 이력서 PDF 링크, 섹션 이동 CTA
- 제약: 문장 수는 최대 2문장, 링크는 2~4개 이내

### Snapshot

- 목적: 이 사이트의 범위를 짧게 알려준다.
- 포함 콘텐츠: 카카오페이 경험 1개, 프로젝트 3개, 공식 이력/증빙 보유 여부 요약
- 제약: 3~4개 항목 이내, 각 항목은 1줄 수준

### Timeline

- 목적: A5 기반의 공식 연혁을 빠르게 훑게 한다.
- 포함 콘텐츠: 학위, 42 Seoul, 부스트캠프, 주요 프로젝트 시기, 카카오페이, 정보처리기사 취득
- 제약: 장문 설명 없이 연도/기간 중심

### Experience

- 목적: 카카오페이 FE 인턴 경험으로 핵심 신뢰도를 만든다.
- 포함 콘텐츠: 회사명, 역할, 기간, 수행 축, 주요 작업 4개, 협업/QA 맥락
- 제약: 가장 상세한 섹션이지만, 각 작업은 `제목 + 2~4개 포인트` 수준으로 유지

### Projects

- 목적: 대표 프로젝트 3개의 성격 차이를 분명히 보여준다.
- 포함 콘텐츠: 프로젝트명, 기간 또는 맥락, 역할, 핵심 구현 포인트, 기술, 저장소 링크
- 제약: 각 프로젝트는 하나의 항목으로 유지하고, 세부 구현을 지나치게 분해하지 않는다.

### Activities

- 목적: 협업과 기록 방식을 보조적으로 보여준다.
- 포함 콘텐츠: 스크럼, 스프린트, 회의록, 개발일지, 멘토링, 문서화 자산
- 제약: 프로젝트처럼 확장하지 않고 보조 섹션으로 유지

### Education & Credentials

- 목적: 학력, 교육/수료, 자격증, 어학을 정리된 공식 이력으로 제공한다.
- 포함 콘텐츠: 학위/트랙/평점/학점, 관련 이수, 부스트캠프, 42 Seoul, 정보처리기사, TOEIC, TOEIC Speaking
- 제약: 해석 문구보다 사실 정리에 집중

### Evidence

- 목적: PDF 원문 증빙을 별도 묶음으로 제공한다.
- 포함 콘텐츠: 이력서, 성적증명서, 졸업증명서, 자격증, 어학 성적표, 부스트캠프 수료증, 42 Seoul 증빙
- 제약: 본문과 시각적으로 분리하고, 문서 종류/발급 의미 중심으로 짧게 표시

### External Links

- 목적: 프로젝트 저장소 등 외부 링크를 정리한다.
- 포함 콘텐츠: 별숲, Sleep Now!, PONG GitHub 링크
- 제약: `spec.md`에 없는 개인 프로필 링크는 넣지 않는다.

## 5. Intro 섹션 설계

Intro는 강한 Hero 랜딩 대신, `짧은 자기소개 + 신뢰 가능한 포트폴리오 안내` 역할로 설계한다.

배치 기준은 다음과 같다.

- 1행: 이름
- 2행: 직무
- 3행: A1을 압축한 1~2문장 소개
- 4행: 주요 링크
- 5행: Experience, Projects, Evidence로 이동하는 CTA

링크 우선순위는 아래처럼 두는 것이 적절하다.

1. Resume PDF
2. Experience로 이동
3. Projects로 이동
4. Evidence 또는 Links로 이동

이번 변경 이후 Intro에서 가능한 링크는 분명해졌다.

- 이력서 PDF는 바로 연결 가능
- 프로젝트 저장소는 개별 섹션 또는 하단 링크에서 연결 가능
- 이메일, 개인 GitHub 프로필, 블로그는 `추가 확인 필요`

피해야 할 방식은 다음과 같다.

- 큰 슬로건 한 줄로 화면 대부분을 쓰는 Hero
- 자격증/점수/배지 나열을 상단에 몰아넣는 방식
- PDF 다운로드 버튼을 다수 배치해 상단이 문서 창고처럼 보이게 하는 방식

## 6. Snapshot / Timeline 설계

### Snapshot

Snapshot은 강점 해석 섹션이 아니라, 이 사이트를 어떻게 읽으면 되는지 알려주는 짧은 요약 구역으로 둔다.

권장 항목은 다음과 같다.

1. 카카오페이 FE 인턴 경험
2. 대표 프로젝트 3개
3. 학위/교육/자격/어학 정리
4. PDF 증빙 및 GitHub 링크 제공

### Timeline

Timeline은 이번 업데이트에서 새로 활용 가능한 요소다. A5 기준으로 짧은 연혁을 보여준다.

권장 배치 항목은 다음과 같다.

- 2019.03 ~ 2024.08: 경희대학교
- 2022.05 ~ 2024.04: 42 Seoul
- 2023.07 ~ 2023.12: 부스트캠프 챌린지/멤버십
- 2024.02 ~ 2024.05: PONG
- 2025.04 ~ 2025.10: 카카오페이
- 2025.09: 정보처리기사

이 섹션은 설명보다 `흐름 정리`가 목적이므로, 긴 문단은 두지 않는다.

## 7. Experience 섹션 설계

Experience는 카카오페이 하나만 다루는 구조로 유지한다. C1-C4를 독립 프로젝트처럼 쪼개지 않고, `카카오페이 FE 인턴` 아래에서 주요 작업을 묶어 보여준다.

권장 구조는 다음과 같다.

- 헤더: 회사명 / 역할 / 기간
- 서브텍스트: 수행 축 요약
- 본문: 주요 작업 4개
- 하단 보조 정보: 협업 방식, QA 대응, 공개 표현 범위 주석

주요 작업 구성안은 아래가 적절하다.

1. 오퍼 고도화
2. 카카오페이포인트 및 운영 백로그
3. 건틀렛 UI 변경
4. Node 16→22 / yarn→pnpm 이관

각 작업의 서술 프레임은 통일한다.

- 무엇을 다룬 작업인지
- 내가 맡은 범위
- 판단 또는 구현 포인트
- 결과 또는 의미

## 8. Projects 섹션 설계

프로젝트는 `대표 1개 + 보조 2개`가 아니라, 이번 버전에서는 `3개 모두 독립 항목`으로 두되 시각적 위계만 조절하는 편이 더 적절하다. 이유는 이제 각 프로젝트가 수행 맥락까지 함께 설명될 수 있기 때문이다.

권장 순서는 아래와 같다.

1. 별숲
2. PONG
3. Sleep Now!

이 순서를 제안하는 이유는 다음과 같다.

- 별숲은 FE 포트폴리오로서 가장 풍부한 구현 설명이 가능하다.
- PONG은 42 Seoul 맥락이 붙으면서 프로젝트 배경이 더 또렷해졌다.
- Sleep Now!는 기술 성격은 명확하지만 기간 정보가 상대적으로 덜 정교하다.

각 프로젝트 카드 또는 섹션은 아래 정보를 포함한다.

- 프로젝트명
- 기간 또는 수행 맥락
- 한 줄 설명
- 역할
- 핵심 포인트 3~5개
- 기술
- GitHub 링크

프로젝트별 배경 표시 기준은 다음을 따른다.

- 별숲: `부스트캠프 멤버십 프로젝트`
- PONG: `42 Seoul 활동 중 프로젝트`
- Sleep Now!: `대학 프로젝트`

## 9. Activities 섹션 설계

Activities는 메인 사례가 아니라 `일하는 방식`을 보완하는 섹션으로 설계한다.

포함 기준은 다음과 같다.

- 카카오페이: 스크럼, 주간회의, Slack, QA, 티켓화
- 별숲: 스프린트, 백로그, 데일리 스크럼, 회의록, 개발일지, 멘토링
- PONG: 문서화, 웹소켓 규약, 협업 자산 정리

표현 방식은 아래가 적절하다.

- 활동 주제명
- 한 줄 설명
- 필요 시 관련 프로젝트/경험 태그

프로젝트 상세처럼 길게 늘리지 않고, `이런 방식으로 협업했다`는 인상을 남기는 수준이면 충분하다.

## 10. Education & Credentials / Evidence / Links 설계

### Education & Credentials

이 섹션은 이번 업데이트에서 가장 크게 강화된 영역이다. `공식 스펙`을 사람이 읽기 쉬운 요약으로 정리한다.

권장 묶음은 다음과 같다.

- Degree: 경희대학교 소프트웨어융합학과 학사, 데이터사이언스트랙, 기간, 학점/평점
- Coursework: 주요 전공 과목
- Training: 부스트캠프 챌린지/멤버십, 42 Seoul Common Core
- Certificate: 정보처리기사
- Language: TOEIC, TOEIC Speaking

표현 원칙은 다음과 같다.

- 날짜와 수치는 정확히 표기한다.
- 해석이나 수식 없이 사실 중심으로 쓴다.
- PDF 링크는 이 섹션 본문보다 `Evidence`로 넘긴다.

### Evidence

Evidence는 원문 증빙 묶음이다. 카드, 아코디언, 간단한 리스트 중 하나로 구현하는 것이 적절하다.

권장 표시 정보는 다음과 같다.

- 문서명
- 문서 종류
- 핵심 확인 포인트 1줄
- 다운로드 링크

문서 묶음 예시는 다음과 같다.

- Resume
- Degree / Transcript
- Training Certificates
- Certification
- Language Scores

주의점은 다음과 같다.

- 본문보다 시각적 강조를 낮춘다.
- PDF가 많으므로 카테고리로 묶는다.
- 원문 공개 전 민감정보 포함 여부는 별도 점검이 필요하다.

### External Links

- Sleep Now! GitHub
- 별숲 GitHub
- PONG GitHub

개인 GitHub 프로필, 이메일, 블로그는 `추가 확인 필요`로 둔다.

## 11. 구현 시 주의사항

- 내부 용어는 공개용 표현으로 풀어쓴다.
- 카카오페이 항목은 팀 성과와 개인 기여를 분리해 적는다.
- Sleep Now!의 정확한 기간은 확정 전까지 모호한 표현을 유지한다.
- 원문 PDF는 공개 전 마스킹 필요 여부를 검토한다.
- `Education & Credentials`와 `Evidence`를 하나의 섹션으로 합치지 않는다.

## 12. 최종 구현 방향 요약

이 사이트는 `소개 → 요약 → 연혁 → 카카오페이 → 프로젝트 3개 → 활동/협업 → 공식 이력 → 증빙 → 외부 링크`의 흐름으로 설계하는 것이 가장 적절하다.  
이번 업데이트 이후에는 특히 `Education & Credentials`와 `Evidence`를 분리하는 것이 중요하다.  
본문은 사람이 빠르게 읽는 정보에 집중하고, PDF와 GitHub는 뒤에서 검증 자료로 제공해야 사이트가 포트폴리오답게 유지된다.
