# 한 페이지 SSG 포트폴리오 기획안

## 문서 목적

이 문서는 [`research.md`](/home/dbwhd/portfolio/research.md)의 분석을 바탕으로, 개인 스펙 마스터 문서를 한 페이지 SSG 포트폴리오로 구현하기 위한 사전 설계안이다.  
지금 단계의 목적은 구조와 콘텐츠 범위를 확정하는 것이며, 구현 상세나 디자인 시스템 제작은 포함하지 않는다.

## 사이트 목표

### 1. 가장 먼저 증명할 것

운영 중인 서비스를 안전하게 바꾸는 프론트엔드라는 인상을 첫 화면과 첫 사례에서 즉시 전달한다.

### 2. 그 다음으로 증명할 것

- 대안 비교와 영향도 분석을 통해 의사결정 품질을 높인다는 점
- 상태 신선도, 에러 처리, 점진 전환 같은 안정성 관점을 설계 단계에서 다룬다는 점
- 협업 과정에서 요구사항과 정책 불확실성을 빠르게 정리한다는 점

### 3. 페이지가 달성해야 하는 사용자 행동

- 채용 담당자: 1분 내 핵심 포지셔닝과 대표 사례를 이해
- 실무 면접관: featured case만 읽어도 문제 정의와 의사결정 수준을 파악
- 팀 리드: 협업 방식과 운영 감각까지 읽고 면접 질문을 구체화

## 페이지 전략

### 왜 SSG 한 페이지가 적합한가

- 포트폴리오의 핵심은 빈번히 갱신되는 데이터가 아니라 검증된 서사 구조이므로 정적 생성이 적합하다.
- 한 페이지는 독자가 맥락 전환 없이 포지셔닝 → 대표 사례 → 보조 사례 → 신뢰 정보까지 선형적으로 읽게 만든다.
- 프로젝트 수 자체를 전시하는 목적이 아니라 일관된 역량을 압축해 증명하는 목적이므로, 다중 라우트보다 한 페이지가 메시지 통제에 유리하다.
- SSG는 로딩 안정성, SEO 기본기, 배포 단순성 측면에서도 목적에 맞다.

### 트레이드오프

- 한 페이지는 각 프로젝트의 전체 서사를 길게 풀기 어렵다.
- 따라서 featured case는 3개로 제한하고, 나머지는 카드/요약형으로 축약해야 한다.
- 추후 상세 페이지 확장 가능성은 열어두되, V1에서는 메시지 일관성을 우선한다.

## 섹션 순서와 목적

### 1. Hero

### 목적

첫 화면 10초 안에 포지셔닝을 각인한다.

### 배치 콘텐츠

- 메인 포지셔닝 1문장
- 이를 뒷받침하는 짧은 증거 문장 2~3개
- 대표 사례로 바로 이동하는 CTA
- 이름/닉네임/직무 표기

### 작성 원칙

- 기술 스택 나열 금지
- 감성적 슬로건보다 "운영 중 서비스", "영향도 분석", "안전장치", "상태/에러 처리" 같은 근거성 표현 우선

### 2. Positioning Summary

### 목적

이 사람이 어떤 기준으로 일하는지 독자가 사례를 읽기 전에 이해하도록 만든다.

### 배치 콘텐츠

- 메인 포지셔닝 1개
- 보조 포지셔닝 2~3개
- 반복되는 강점 패턴 3~5개
- 타깃 독자별 핵심 메시지를 직접 노출하지는 않더라도, 그 메시지를 압축한 요약 문장

### 표현 방식

- 문장형 요약 + 짧은 원칙 카드
- "대안 비교", "영향도 분석", "정책 UX", "freshness", "에러 처리", "문서화" 같은 축을 노출

### 3. Featured Case Studies

### 목적

포트폴리오의 핵심 신뢰 구간이다.  
각 사례는 기능 소개가 아니라 문제 해결 방식과 안전한 변경 능력을 입증해야 한다.

### 공통 템플릿

- 배경/문제
- 목표
- 핵심 제약 또는 리스크
- 고려한 대안 또는 선택 기준
- 내가 한 일
- 결과/효과
- 근거 또는 증빙 링크
- 추가 확인 필요 항목이 있으면 별도 표시

### Featured Case 선정안

#### Case 1. 카카오페이 통합 마케팅 어드민 오퍼 고도화

- 가장 깊게 서술
- 핵심 포인트:
  - 메시지 타입 도입에 따른 구조 확장
  - 변수 폼/템플릿/임시저장/기존 오퍼 불러오기 확장
  - 예약 현황 캘린더 UI 프로토타입
  - GET→POST 이관 영향도 분석
  - 정책 확인 모달
  - QA/요구사항 확정 과정
  - V2 분리로 점진 전환

#### Case 2. 별숲 3D 밤하늘 UX + 데이터 신선도 중심 설계

- 중간 깊이 서술
- 핵심 포인트:
  - OrbitControls 채택 이유
  - 더블클릭 시점 이동 대안 비교와 최종 선택
  - 상태 코드 기반 API 처리 규칙
  - React-Query 공통화
  - History API vs Recoil 비교 후 freshness 우선 결정

#### Case 3. 카카오페이 Node 16→22 업그레이드 & yarn→pnpm 마이그레이션

- 상대적으로 압축하되 featured로 유지
- 핵심 포인트:
  - 로컬/CI 관점 검증
  - 이관 과정에서 발생 가능한 이슈 확인 및 해결
  - 팀 표준화 및 지식 공유 기여
  - 성과 수치는 공개 가능 범위 내에서만 사용

- 강조 포인트: 디자인 시스템 업그레이드, 에러 페이지 교체, 서비스 종료 대응, 요구사항 빠른 확정

- 한 줄 요약: CSS vs SVG 실제 비교를 통해 반응형 제약 안에서 최적안 도출
- 강조 포인트: 완성도와 반응형 왜곡 사이의 트레이드오프

#### PONG! PONG! PONG!

- 한 줄 요약: 모달 UI/API, Three.js 그래픽, 로컬 WebSocket, 게임 로직을 맡아 서비스 기능과 플레이 경험을 연결

- 한 줄 요약: 센서 기반 예측과 notification 중재를 결합한 React Native 프로젝트

## 구현 우선순위

### V1 범위

- 한 페이지 SSG 구조
- Hero / Positioning / Featured Cases / Additional Projects / About / Links
- featured case 3개 콘텐츠 구조화
- 핵심 메타데이터
- 기본 반응형 대응

### 이후 확장 범위

- featured case 상세 페이지 분리
- 화면 캡처/다이어그램 추가
- 필터링 또는 축약/펼치기 인터랙션
- 영문 버전
- 블로그/회고 연결

## 디자인/UX 원칙

- 첫 화면은 시각 장식보다 메시지 전달 속도를 우선한다.
- 시선 흐름은 Hero에서 바로 대표 사례로 내려가도록 설계한다.
- 각 사례는 "문제 → 판단 → 결과"가 한 화면 안에서 읽히게 구성한다.
- 스택 배지는 최소화하고, 판단 근거와 리스크 설명을 더 크게 배치한다.
- 시각적 강조는 "안전한 변경", "대안 비교", "freshness", "에러 처리" 같은 키워드에 집중한다.
- 카카오페이 사례는 신뢰감 있는 정보 구조가 중요하므로 과도한 장식보다 정돈된 레이아웃이 적합하다.
- 별숲과 PONG의 시각성은 포트폴리오 전체 분위기를 바꾸는 메인 톤이 아니라 보조 포인트로 사용한다.

## 애니메이션 사용 원칙

- 정보 이해를 방해하지 않는 범위에서만 사용한다.
- Hero 진입, 섹션 전환, 카드 등장 정도의 가벼운 모션만 허용한다.
- featured case 본문에는 과도한 패럴랙스나 긴 스크롤 연출을 넣지 않는다.
- 모션은 "몰입감"보다 "정보 계층 강조"를 위한 수단이어야 한다.
- 3D 사례를 소개하더라도 사이트 전체를 3D 포트폴리오로 만들 필요는 없다.
