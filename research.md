# 포트폴리오 사전 기획 리서치

## 문서 목적

이 문서는 [`spec.md`](/home/dbwhd/portfolio/spec.md)를 단일 source of truth로 삼아, 한 페이지 SSG 포트폴리오를 설계하기 전에 어떤 메시지를 전면에 내세워야 하는지 분석한 결과다.  
목적은 프로젝트를 많이 했다는 인상을 주는 것이 아니라, 운영 중인 서비스를 안전하게 바꾸는 프론트엔드라는 일관된 인상을 구조적으로 증명하는 데 있다.

## 핵심 포지셔닝

### 메인 포지셔닝

운영 중인 서비스에서 정책·연동·배포 리스크를 먼저 읽고, 영향도 분석과 UX 안전장치로 변경을 안정적으로 완수하는 프론트엔드

### 보조 포지셔닝

1. 구현 전에 대안을 실제로 비교하고 한계를 검증해 팀 의사결정 품질을 높이는 프론트엔드
2. 상태 신선도, 에러 처리, API 흐름 공통화처럼 장기 유지보수에 직접 연결되는 설계를 우선하는 프론트엔드
3. 기록과 문서화를 통해 협업 비용을 줄이고, 불확실한 요구사항을 빠르게 정리하는 프론트엔드

## 타깃 독자별 핵심 메시지

### 채용 담당자

- 단순 기능 구현자가 아니라 운영 리스크를 줄이는 방향으로 일하는 사람이라는 점이 빠르게 보여야 한다.
- 카카오페이 사례를 통해 실제 서비스 환경에서 정책, 운영, QA, 시스템 변경을 다뤘다는 점이 먼저 읽혀야 한다.
- 학력, 자격, 어학, 사용 기술은 신뢰 보강 요소로는 유효하지만 메인 메시지가 되어서는 안 된다.

### 실무 프론트엔드 면접관

- 이 포트폴리오는 "무엇을 만들었는가"보다 "왜 그 구조를 택했고 어떤 대안을 버렸는가"를 보여줘야 한다.
- 오퍼 고도화의 V2 분리, API GET→POST 이관 영향도 분석, 별숲의 History API vs Recoil 비교는 기술 판단력을 설명하는 핵심 근거다.
- 상태 신선도, 401 재발급 흐름, 처리 공통화, 정책 확인 모달 같은 항목을 통해 프론트엔드를 UI가 아니라 시스템 경계 관리 관점으로 다룬다는 점을 전달해야 한다.

### 협업 관점의 팀 리드

- 요구사항이 불명확할 때 빠르게 확정하고, QA와 운영 이슈를 문서화/티켓화해 재작업 비용을 줄이는 사람이라는 점이 보여야 한다.
- 별숲의 운영 자산, 카카오페이의 회의/슬랙/QA 대응, PONG의 웹소켓 규약 문서화는 협업 신뢰도를 보강한다.
- 팀 성과를 자신의 성과처럼 포장하지 않고 개인 기여 경계를 분명히 하는 태도도 중요한 신뢰 요소다.

## 가장 강한 사례 우선순위

### 1위. 카카오페이 통합 마케팅 어드민 오퍼 고도화

이 사례를 가장 앞에 둬야 한다. 이유는 세 가지다.

- 운영 중인 서비스에서 실제로 발생하는 위험을 가장 입체적으로 보여준다.
- 단순 화면 구현이 아니라 데이터 구조 확장, 폼 구조 변경, API 이관 영향도 분석, 정책 확인 모달, QA 대응까지 포함되어 있다.
- "안전하게 바꾸는 프론트엔드"라는 핵심 포지셔닝과 가장 직접적으로 맞닿아 있다.

이 사례가 중요한 이유는 결과보다 과정에 있다. 특히 V2 분리로 대규모 API 변경을 점진 전환 구조로 다룬 점, 동일 API 사용 지점을 탐색해 잠재 장애를 사전에 공유한 점, 예외 정책을 모달로 강제해 운영 사고를 예방한 점은 문제 정의와 리스크 관리 역량을 함께 보여준다.  
한 페이지 포트폴리오에서 가장 깊게 다뤄야 할 사례다.

### 2위. 별숲 3D 밤하늘 UX + 데이터 신선도 중심 설계

두 번째로 배치할 가치가 높다. 이유는 다음과 같다.

- 카카오페이 사례가 운영/정책/리스크 관리 축이라면, 별숲은 복잡한 인터랙션 환경에서의 기술 판단력을 보여준다.
- OrbitControls 선택, 더블클릭 시점 이동 대안 비교, History API vs Recoil 비교, 상태 코드 기반 API 처리 규칙, React-Query 공통화가 모두 "대안 검토 후 선택"이라는 패턴을 증명한다.
- 3D 프로젝트라는 시각적 강점이 있어 한 페이지 포트폴리오에서 시선을 끄는 역할도 가능하다.

다만 이 사례의 핵심은 3D 그 자체가 아니라, 복잡한 UX에서도 데이터 freshness와 상태/에러 처리를 우선했다는 점으로 정리해야 한다.  
즉 "화려한 프로젝트"로 소개하면 메시지가 흐려지고, "복잡한 환경에서도 안정성을 우선한 설계 사례"로 다뤄야 한다.

### 3위. 카카오페이 Node 16→22 업그레이드 & yarn→pnpm 마이그레이션

세 번째 featured case로 적합하다.

- 제품 UI 사례가 아니라 개발환경/빌드 체계 변경을 다룬다는 점에서 포트폴리오의 폭을 넓혀 준다.
- 운영 중 서비스에서 런타임과 패키지 매니저를 바꾸는 작업은 실패 비용이 크기 때문에, 안전한 검증과 표준화 기여라는 메시지와 연결된다.
- 오퍼 고도화가 사용자/운영자 흐름, 별숲이 상태 설계라면, 이 사례는 팀 DX와 배포 기반 안정성을 보강한다.

다만 수치가 구체적으로 공개되어 있지 않다. 따라서 "정량 성과"보다 "검증 방식과 팀 내 공유"를 중심으로 다뤄야 하며, 내부 지표 공개 가능 범위는 추가 확인이 필요하다.

### 후순위 사례 판단

#### 카카오페이 포인트/운영 백로그

운영 안정성, 요구사항 확정, 기존 동작 보존 원칙을 보여주는 데 유효하다.  
하지만 하나의 깊은 서사라기보다 여러 운영성 작업 묶음에 가깝기 때문에 featured case보다는 보조 카드가 적합하다.

#### 카카오페이 건틀렛 쿠폰 UI 변경

CSS vs SVG 대안 비교가 명확해 짧은 카드형 사례로 좋다.  
다만 영향도와 문제 복잡도가 오퍼 고도화보다 작아 메인 사례로 전면 배치하기에는 상대적으로 약하다.

#### Sleep Now!

센서, 예측, Foreground Service, notification 중재라는 주제가 독특해 기술 폭을 보여주는 데는 강하다.  
하지만 현재 포트폴리오 목표인 "운영 중인 서비스를 안전하게 바꾸는 프론트엔드"와 직접 연결되는 증거는 카카오페이보다 약하다. 또한 95% 정확도는 README 기반 기록이므로 맥락 설명 없이 전면 배치하면 과장처럼 보일 수 있다.

#### PONG! PONG! PONG!

WebSocket, 게임 로직, Three.js 그래픽 등 구현 폭은 넓다.  
다만 spec 기준으로는 문제 정의, 대안 비교, 결과 검증의 서사가 상대적으로 약하다. 따라서 메인 케이스보다 보조 프로젝트로 두는 편이 메시지 일관성에 유리하다.

## 한 페이지 포트폴리오에서 전면에 내세워야 할 요소

### 전면 배치

- 운영 중 서비스의 리스크를 줄인 사례
- 대안 비교 후 선택한 의사결정 과정
- 영향도 분석, 점진 전환, UX 안전장치, 데이터 신선도, 에러 처리
- 팀과의 조율 방식과 불확실성 해소 방식
- 개인 기여 범위가 분명한 설명

이 요소들은 "프론트엔드 구현자"가 아니라 "서비스를 안전하게 변화시키는 사람"이라는 인상을 만든다.  
특히 첫 화면과 featured case 영역은 이 축으로만 설계하는 것이 맞다.

### 축약해야 할 요소

- 기술 스택 나열
- 학력/자격/어학의 상세 설명
- 프로젝트별 모든 기능 목록
- 3D, 게임, 센서 기술 자체의 흥미 요소

이 항목들은 완전히 불필요하지 않지만, 앞에 나오면 포지셔닝이 "다양한 걸 해본 사람" 수준으로 퍼진다.  
한 페이지에서는 근거 보강용으로 뒤로 보내거나 한 줄 수준으로 압축하는 편이 적합하다.

### 제외해도 되는 요소

- 프로젝트별 세세한 UI 변경 목록의 장황한 나열
- 구현 기술에 대한 교과서식 설명
- 문서에 근거가 없는 수치형 성과 추정

특히 성과 수치를 임의로 보강하면 신뢰가 손상된다.  
한 페이지 특성상 제외 기준은 "포지셔닝 증명에 직접 기여하는가"로 잡아야 한다.

## 반복적으로 드러나는 강점 패턴

### 1. 먼저 바꾸지 않고 먼저 비교한다

CSS vs SVG, OrbitControls vs Raycasting vs 카메라 이동, History API vs Recoil처럼 대안을 실제로 검토하고 한계를 설명한 뒤 선택한다.  
이 패턴은 단순 취향이 아니라 리스크를 줄이는 문제 해결 방식으로 읽힌다.

### 2. 상태와 흐름의 안정성을 UX보다 앞단에서 설계한다

별숲의 freshness 우선 설계, 401 재발급 흐름, React-Query 공통화, 오퍼 고도화의 V2 분리, API 이관 영향도 분석은 모두 "보이는 화면"보다 "깨지지 않는 흐름"을 먼저 본다는 증거다.

### 3. 정책과 운영 리스크를 UI에서 흡수한다

정책 확인 모달, 캘린더 기반 예약 현황, 운영성 변경 시 기존 동작 유지 원칙은 프론트엔드가 정책 리스크를 마지막 단계에서 흡수하는 역할을 한다는 관점을 보여준다.

### 4. 불확실성을 방치하지 않고 협업 구조로 정리한다

슬랙/회의/기획서 댓글/QA 티켓화, 별숲의 기록 기반 운영, PONG의 통신 규약 문서화는 모두 구현 이전 혹은 구현 중에 해석 차이를 줄이는 방식이다.

### 5. 시각적 복잡도보다 사용 맥락을 우선한다

별숲의 3D도 몰입감 자체보다 자연스러운 관찰 경험을 위해 설계되었고, 카카오페이 캘린더 UI도 학습 비용을 낮추는 방향으로 선택되었다.  
즉 "멋있어 보이는 UI"보다 "의도한 행동을 안정적으로 유도하는 UI"를 지향한다.

## 사이트 콘텐츠로 변환할 때의 리스크

### 1. 카카오페이 사례가 내부 도메인 용어에 갇힐 위험

"오퍼", "카페인", "건틀렛" 같은 내부 맥락을 그대로 노출하면 외부 독자가 빠르게 이해하기 어렵다.  
따라서 도메인 용어를 유지하되, 반드시 일반 언어로 병기해야 한다.

### 2. 팀 프로젝트 사례에서 개인 기여가 흐려질 위험

별숲, PONG, Sleep Now는 팀 혹은 프로젝트 단위 서술 비중이 높다.  
사이트로 옮길 때는 "프로젝트 설명"보다 "내가 맡은 판단/구현/정리 범위"를 먼저 보여줘야 한다.

### 3. 기술 다양성이 오히려 포지셔닝을 흐릴 위험

3D, 게임, 모바일 센서, 마이그레이션 등 주제가 넓다.  
정리 없이 펼치면 강점이 넓어 보이기보다 초점이 없는 인상으로 바뀔 수 있다.  
그래서 모든 사례를 "리스크 관리와 의사결정"이라는 공통 축 아래 재배치해야 한다.

### 4. 정량 근거 부족 구간에서 과장처럼 보일 위험

Node/pnpm 이관의 성능 개선, 카카오페이 운영성 개선의 효과는 서술상 유의미하지만 공개 가능한 정량 수치가 없다.  
따라서 결과를 크게 부풀리기보다, 어떤 위험을 줄였는지와 어떤 방식으로 검증했는지를 중심으로 써야 한다.

### 5. 시각 자료가 없으면 사례 이해 속도가 떨어질 위험

캘린더 UI, 3D 밤하늘, PONG, Sleep Now는 화면이나 흐름 이미지가 있으면 전달력이 높다.  
하지만 현재 spec에는 이미지 자산 정보가 없다. 구현 단계 전, 공개 가능한 캡처/도식 보유 여부 확인이 필요하다.

## 정보 공백 및 추가 확인 필요 항목

- 카카오페이 인턴 기간의 정확한 시작일/종료일
- 카카오페이 사례에서 외부 공개 가능한 화면 캡처 또는 도식 자료 유무
- Node/pnpm 이관에서 공개 가능한 정량 지표 범위
- 오퍼 고도화에서 공개 가능한 결과 표현 수준
- 카카오페이 내부 용어를 외부 포트폴리오에서 어느 수준까지 그대로 써도 되는지
- 별숲, PONG, Sleep Now의 배포 링크 또는 시연 영상 유무
- 영문명 표기 방식
- GitHub 외에 링크드인, 블로그, 회고 글 등 추가 증빙 링크 유무
- 학력/자격/어학 정보를 한 페이지 내 어디까지 노출할지에 대한 선호
- Sleep Now의 95% 정확도 수치에 대한 실험 조건 설명 가능 여부
