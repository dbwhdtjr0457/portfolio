# 포트폴리오 방향 분석 리서치

본 문서는 [`spec.md`](/home/dbwhd/portfolio/spec.md)를 단일 source of truth로 삼아, 포트폴리오 사이트의 방향성과 정보 우선순위를 정리한 분석 문서다. 문서에 없는 사실은 추가하지 않았으며, 불명확한 항목은 `추가 확인 필요` 또는 `불확실`로 표기했다.

## 1. 이 포트폴리오의 적절한 톤 정의

이 포트폴리오의 적절한 톤은 `담백한 자기소개형`을 기본으로 하되, 형식은 `정돈된 웹 이력서형 포트폴리오`에 더 가깝다. 즉, 첫 화면에서 짧게 자기소개를 한 뒤, 이후에는 `경험 1개 / 프로젝트 1개 / 활동 1개` 단위로 단순하고 읽기 쉽게 나열하는 방향이 적절하다.

그렇게 봐야 하는 이유는 `spec.md`의 정보가 하나의 강한 메시지로 압축되기보다, 카카오페이 실무 경험, 팀 프로젝트, 기술 범위, 협업 기록이 각각 다른 결로 쌓여 있기 때문이다. 따라서 이를 하나의 강점 서사로 묶기보다, `항목별로 정리된 소개`로 보여주는 편이 문서 성격과 더 잘 맞는다.

- 너무 브랜딩형이 되면 어색한 이유
  - A1의 포지셔닝 문장은 분명하지만, 전체 문서의 무게중심은 슬로건보다 `의사결정 방식과 사례 근거`에 있다.
  - 카카오페이 오퍼 고도화(C2), 카페인/운영 백로그(C3), Node/pnpm 이관(C1)은 모두 실무 맥락과 운영 안정성이 핵심인데, 이를 강한 카피 중심으로 덮으면 실제 장점이 가벼워 보일 수 있다.
  - 별숲(C5), PONG(C6), Sleep Now!(C7)도 기술적 폭을 보여주지만, "창의적이고 화려한 개발자" 한 방향으로 묶기에는 성격이 다르다.

- 너무 문서형이 되면 밋밋해지는 이유
  - 완전히 이력서 문장만 나열하면, 카카오페이 실무 경험과 별숲/Sleep Now!/PONG의 성격 차이가 잘 드러나지 않는다.
  - 각 항목에 `무엇을 한 경험인지` 정도의 짧은 설명은 필요하다. 다만 이것이 별도의 강점 해석 섹션으로 커질 필요는 없다.
  - 즉, 평평한 나열은 유지하되 각 묶음의 첫 문장에서만 인상을 남기는 정도가 적절하다.

- 적절한 균형점
  - 첫 화면에서는 직무와 짧은 소개만으로 방향을 잡고, 본문은 복잡한 해석 없이 `경력 1개`, `프로젝트 1개씩`, `활동/기술/학력` 순으로 정리하는 편이 맞다.
  - 강점은 별도 섹션에서 전면화하지 말고, 각 경험 설명 안에서 자연스럽게 읽히게 두는 편이 적절하다.
  - 즉, 구조는 단순해야 하고, 어필은 소개 문장과 각 항목의 짧은 설명 안에만 제한해야 한다.

근거: A1, A3, B1-B4, C1-C7, D1-D15

## 2. 사이트가 전달해야 하는 핵심 인상

### 한 줄 인상

실무 경험과 프로젝트를 과장 없이 정리해 보여주되, 각 경험의 성격이 분명하게 읽히는 프론트엔드 포트폴리오.

### 보조 인상

- 카카오페이 실무 경험이 가장 먼저 읽히는 구조.
- 프로젝트마다 성격과 기여 범위가 구분되어 보이는 구조.
- 기술 나열보다 경험 단위 소개가 우선인 구조.

이 인상 조합이 적절한 이유는, 채용 담당자는 먼저 `무슨 경험이 있는지`를 빠르게 읽고 싶고, 실무 면접관은 각 프로젝트의 성격과 기여 범위를 구분해서 보고 싶기 때문이다. 따라서 이 사이트는 강점 해석보다 `경험 단위 정리`가 먼저여야 한다.

근거: A1-A3, C2-C5, E

## 3. 상단 소개에서 드러나야 할 내용

상단 소개는 과장된 슬로건보다 `짧고 바로 이해되는 자기소개`가 중심이어야 한다. 이 영역에서 반드시 전달되어야 하는 것은 다음과 같다.

- 이름
- 직무: 프론트엔드 개발자
- 짧은 소개 문장: A1의 포지셔닝을 과장 없이 압축한 1~2문장
- 현재 사이트가 무엇을 담고 있는지 보여주는 짧은 안내
- 외부 링크

다만 외부 링크의 구체 범위는 구분이 필요하다.

- 프로젝트 증빙 링크는 F 섹션에 근거가 있어 상단 또는 상단 인접 영역에서 노출 가능하다.
- 일반적인 연락처 링크, 이력서 링크, GitHub 프로필 링크는 `spec.md`에 직접 기재되어 있지 않으므로 `추가 확인 필요`다.

상단에서 어필해야 하는 수준은 `어떤 배경의 프론트엔드인지 알 수 있는 정도`까지다. 그 이후부터는 담백하게 내려가야 한다. 예를 들어 A1의 세 문장을 모두 전면에 내세우기보다, 이를 짧게 정리한 뒤 바로 경력과 프로젝트 목록으로 내려가는 편이 전체 톤과 맞다.

또한 상단 소개는 `기술 키워드 나열`보다 `경험 범위 요약`이 앞에 와야 한다. 기술 스택은 뒤에서 정리해도 되며, 상단에서는 실무 경험과 프로젝트 경험이 함께 있는 포트폴리오라는 점만 분명하면 충분하다.

근거: A1-A3, A2, F

## 4. 문서 전체에서 반복적으로 드러나는 강점 패턴

이 항목은 포트폴리오에서 별도 강조 섹션으로 드러내기보다, 작성자가 정리 기준으로만 참고하는 수준이 적절하다. 즉, 아래 내용은 사이트에서 `강점 패턴`이라는 제목으로 노출하기보다, 각 경험 설명을 쓸 때의 내부 기준으로만 남겨두는 편이 맞다.

### 1. 실무 변경과 운영 맥락이 자주 등장한다

카카오페이의 오퍼 고도화(C2), 운영 백로그(C3), Node/pnpm 이관(C1)은 모두 운영 중인 환경의 변경과 연결되어 있다. 따라서 카카오페이는 여러 카드로 쪼개기보다 `실무 경험 1개`로 묶는 편이 자연스럽다.

### 2. 프로젝트마다 기술적 성격이 분명히 다르다

별숲(C5)은 3D UX와 상태 설계, PONG(C6)은 실시간 게임과 WebSocket, Sleep Now!(C7)는 React Native와 센서/알림 중재가 중심이다. 따라서 프로젝트들은 `프로젝트당 1개 항목`으로 나누는 편이 읽기 쉽다.

### 3. 세부 강점보다 항목별 맥락 구분이 더 중요하다

문서에는 대안 비교, 운영 리스크 대응, 상태 설계 같은 요소가 반복되지만, 이를 먼저 묶어 설명하면 경험 자체가 뒤로 밀린다. 현재 방향에서는 이런 요소를 별도 강점 섹션으로 뽑지 않는 편이 적절하다.

### 4. 협업/기록은 별도 독립 항목보다 보조 정보에 가깝다

E 섹션의 기록과 협업 자산은 중요하지만, 메인 전개를 이끌기보다 각 경험 또는 후반부 활동 섹션에서 보조적으로 정리하는 편이 자연스럽다.

근거: C1-C7, E

## 5. 대표 사례와 보조 사례 우선순위

이 항목은 `대표 사례 몇 개를 뽑는 방식`보다 `경험/프로젝트를 몇 개의 묶음으로 보여줄지`를 정하는 기준으로 해석하는 편이 현재 요청과 맞다.

### 우선순위가 높은 묶음

#### 1. 카카오페이

카카오페이는 하나의 대표 경력 항목으로 묶는 편이 적절하다.

- B1 기준으로 이미 수행 축이 한 묶음으로 정리되어 있다.
- C1, C2, C3, C4를 각각 독립 프로젝트처럼 앞세우기보다, `카카오페이 FE 인턴 경험` 아래에서 주요 작업들을 나열하는 편이 더 단순하고 읽기 쉽다.
- 이 방식이 사용자 요청인 `카카오페이 하나`라는 형식과 가장 잘 맞는다.

#### 2. 별숲(ByeolSoop)

별숲은 프로젝트 1개 항목으로 독립 배치하는 편이 적절하다.

- 3D 밤하늘 UX, 카메라 이동, API 처리 규칙, React-Query 공통화 등 설명할 내용은 많지만, 하나의 프로젝트 소개 안에서 정리 가능하다.
- 별숲 내부의 세부 판단을 다시 여러 사례로 쪼개지 않는 편이 사이트 전체 톤과 맞다.

#### 3. Sleep Now!

Sleep Now!도 프로젝트 1개 항목으로 배치하는 편이 적절하다.

- React Native, 센서, 예측, 알림 중재라는 프로젝트 성격이 뚜렷하다.
- 성능 수치와 개선 방향은 있더라도, 이를 별도 카드로 확장하기보다 프로젝트 설명 안에서 함께 다루는 편이 단순하다.

#### 4. PONG! PONG! PONG!

PONG 역시 프로젝트 1개 항목으로 배치하는 편이 적절하다.

- 대시보드 모달, Three.js 그래픽, 로컬 WebSocket, 게임 계산 로직 등 역할 범위가 문서에 정리되어 있다.
- 여러 기술 요소가 있어도 한 프로젝트 설명으로 묶는 편이 더 읽기 쉽다.

### 보조 정보로 처리할 항목

- Node 16→22 / yarn→pnpm 이관(C1)
- 오퍼 고도화(C2)의 세부 포인트
- 운영 백로그(C3)
- 건틀렛 UI 변경(C4)
- 협업/문서화 자산(E)

이 항목들은 독립 프로젝트로 늘리기보다, `카카오페이 소개 내부의 작업 목록` 또는 `후반부 보조 정보`로 처리하는 편이 적절하다.

정리하면, 이 포트폴리오는 `카카오페이 1개 + 별숲 1개 + Sleep Now! 1개 + PONG 1개`의 중심 구조가 가장 자연스럽고, 세부 사례는 각 항목 내부 설명으로 들어가는 편이 맞다.

근거: C1-C7, D1-D15

## 6. 정보구조 관점의 우선순위

읽는 흐름 기준으로는 아래 순서가 적절하다.

### 1. 소개

이름, 직무, 짧은 소개가 가장 먼저 와야 한다. 여기서는 강점 해석보다 `어떤 경험 구성을 가진 프론트엔드인지`가 빠르게 파악되면 충분하다.

### 2. 경력

카카오페이 인턴 경험(B1)은 가장 먼저 오는 단일 경력 항목이 적절하다. 여기서 C1, C2, C3, C4를 세부 작업 목록으로 묶어 보여주는 흐름이 맞다.

### 3. 프로젝트

프로젝트는 `별숲`, `Sleep Now!`, `PONG! PONG! PONG!`를 각각 하나의 항목으로 두는 편이 적절하다. 프로젝트 내부의 세부 구현 포인트는 각 항목 안에서 짧게 정리하고, 프로젝트를 다시 여러 사례로 쪼개지는 않는 편이 좋다.

### 4. 기술

기술 스택은 필요하지만, 이 포트폴리오에서는 앞쪽에 크게 배치할 이유가 약하다. A2는 넓고 좋지만, 차별점은 기술명보다 문제 해결 패턴에 있기 때문이다. 따라서 기술은 중후반부에서 `사례를 읽고 난 뒤 정리해주는 역할`이 적절하다.

### 5. 활동 / 협업 방식

문서화, 스크럼, 멘토링, Ground Rule, 웹소켓 규약 문서화(E)는 분명 강점이지만, 처음부터 전면에 두면 너무 설명적으로 보일 수 있다. 사례를 본 뒤 "이런 방식으로 팀에서 일하는구나"를 보강하는 위치가 적절하다.

### 6. 학력 / 자격 / 어학

공식 스펙(A4)은 신뢰도를 보완하지만, 이 포트폴리오의 첫 인상을 결정할 정보는 아니다. 뒤쪽에 간결하게 정리하는 편이 자연스럽다.

즉, 이 사이트는 `소개 → 카카오페이 경력 1개 → 프로젝트 3개 → 기술 → 활동/협업 → 공식 스펙`의 흐름이 가장 읽기 쉽다.

근거: A1-A4, B1-B4, C1-C7, E

## 7. 너무 강조하면 안 되는 요소 / 너무 약하면 안 되는 요소

### 너무 강조하면 안 되는 요소

- 하나의 슬로건으로 전체를 지배하는 브랜딩 문장
  - A1의 포지셔닝은 강하지만, 전체 문서는 그보다 사례 기반 해석에 가깝다.
- 강점 패턴을 별도 섹션으로 전면에 배치하는 구성
  - 현재 요청은 강점 해석보다 단순한 경험 나열에 가깝기 때문에, `대안 비교`, `운영 리스크 대응` 같은 항목을 앞에서 묶어 설명하는 방식은 과해질 수 있다.
- 카카오페이 안의 작업을 각각 독립 프로젝트처럼 분리하는 구성
  - C1-C4를 따로 떼면 경력보다 사례 모음처럼 보일 수 있다.
- 기술 키워드 대량 노출
  - A2의 키워드는 풍부하지만, 먼저 나열하면 일반적인 기술 스택 소개처럼 보일 수 있다.

### 너무 약하면 안 되는 요소

- 카카오페이 실무 사례
  - 운영 중인 서비스 경험과 리스크 대응 감각은 이 포트폴리오의 신뢰도를 크게 만든다.
- 프로젝트별 구분
  - 별숲, Sleep Now!, PONG는 각각 기술 성격이 달라 분리되어 보여야 한다.
- 각 항목의 짧은 설명
  - 단순 제목 나열만으로는 무엇을 한 프로젝트인지 전달되지 않는다.
- 협업과 문서화
  - 메인 전면은 아니어도 후반부에서 빠지면 전체 일 방식이 보이지 않는다.

### 균형이 필요한 지점

- 자기소개 문장
  - 한 줄 요약은 필요하지만, 사이트 전체를 지배하는 광고 문구처럼 보이면 어색하다.
- 카카오페이 내부 작업 소개
  - 하나의 경력 항목으로 묶되, C1-C4의 성격 차이는 짧게라도 보이게 해야 한다.
- 팀 프로젝트 설명
  - 별숲, PONG, Sleep Now!는 각각 한 항목으로 묶되 `내 역할 범위`가 흐려지지 않게 조절해야 한다.

근거: A1-A3, C1-C7, E

## 8. 리스크와 추가 확인 필요 사항

### 리스크

- 내부 용어 노출 리스크
  - `카페인`, `건틀렛`, `FIT`, `오퍼`, `메시지 통합 Phase 1` 같은 표현은 내부 맥락이 없으면 읽는 사람이 이해하기 어렵다.
  - 공개 포트폴리오에서는 설명 없이 노출할 경우 정보 전달력이 떨어질 수 있다.

- 팀 성과와 개인 기여 혼동 리스크
  - C1의 빌드 개선 체감 성과, C2의 프로젝트 전체 효율 향상, C5-C7의 팀 프로젝트 결과는 `팀 결과`와 `개인 기여`를 섞어 쓰면 과장처럼 보일 수 있다.
  - spec의 운영 원칙(0번 섹션)대로 분리 표기가 필요하다.

- 수치 부재 또는 공개 범위 불명확 리스크
  - C1은 내부 지표 언급이 있으나 구체 수치가 없다.
  - C7은 정확도 기록이 있으나 실험 조건, 표본 범위, 실제 서비스 품질과의 관계는 추가 설명이 없다.
  - 따라서 수치를 전면에 내세우기보다, 수치가 확실한지 먼저 확인해야 한다.

- 프로젝트 간 비중 불균형 리스크
  - 카카오페이는 한 회사 경험 안에 여러 작업이 있고, 프로젝트들은 각 1개 항목씩이므로 길이 차이가 크게 벌어질 수 있다.
  - 단순 나열 구조를 택하더라도 항목 길이 조절이 필요하다.

- 공개 증빙 한계 리스크
  - F 섹션에는 GitHub 링크가 있지만, 카카오페이 사례는 공개 저장소 근거가 없다.
  - 회사 경험을 어느 수준까지 구체적으로 공개 가능한지 `추가 확인 필요`다.

### 추가 확인 필요 사항

- 상단과 하단에 둘 수 있는 공식 외부 링크 범위
  - 프로젝트 GitHub 외에 개인 GitHub 프로필, 연락처, 이력서 링크 공개 여부는 `추가 확인 필요`.

- 카카오페이 사례의 공개 가능 표현 범위
  - 내부 명칭, 화면, 정책 내용, 성과 표현 수위를 어디까지 공개할 수 있는지 `추가 확인 필요`.

- 각 사례의 시간 정보
  - 카카오페이 인턴은 약 7개월로 적혀 있으나, 별숲/PONG/Sleep Now!의 정확한 기간은 본 문서 기준으로 `불확실`.

- 프로젝트 표시 순서
  - 별숲, Sleep Now!, PONG의 노출 순서를 어떤 기준으로 둘지 최종 정리 필요하다.
  - 현재 문서만으로는 기술 난도, 최신성, 지원 포지션 우선순위 중 무엇을 기준으로 할지 `불확실`하다.

근거: 0, C1-C7, F

## 9. 최종 요약

이 포트폴리오는 강한 브랜딩형보다, 정돈된 웹 이력서형 포트폴리오에 가깝게 가는 편이 맞다.  
상단에서는 이름, 직무, 짧은 소개만 보여주고, 이후에는 해석보다 나열 중심으로 내려가는 구조가 적절하다.  
특히 카카오페이는 `경력 1개`로 묶고, 내부 작업인 C1-C4는 그 안의 주요 내용으로 정리하는 편이 자연스럽다.  
프로젝트는 `별숲`, `Sleep Now!`, `PONG! PONG! PONG!`를 각각 하나의 항목으로 나누는 구성이 맞다.  
강점 패턴을 별도 섹션으로 전면화하기보다, 각 항목의 짧은 설명 안에서만 자연스럽게 읽히게 두는 편이 현재 목표와 맞다.  
정보 흐름은 `소개 → 카카오페이 → 프로젝트들 → 기술 → 활동/협업 → 공식 스펙` 순서가 가장 읽기 쉽다.  
전체적으로는 객관적이고 단순한 구조를 유지하되, 항목 구분만으로 경험의 성격이 드러나는 포트폴리오가 적절하다.
