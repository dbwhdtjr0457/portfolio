# C2. 카카오페이 — 통합 마케팅 어드민 “오퍼” 고도화(메시지 통합 Phase 1)

## 1) 배경/문제

- 마케터가 쿠폰/메시지 발송을 각각 다른 시스템에서 수행해야 해서 작업 시간이 길고, 시스템 왕복으로 휴먼 에러 위험이 존재.
- 이를 통합 마케팅 툴로 일원화해 업무 효율을 높이는 프로젝트가 진행됨.

## 2) 목표

- 마케터의 발송 업무 흐름을 단일 시스템에서 처리 가능하게 만들고,
- 스케줄 가시성 및 정책 기반 안전장치를 통해 운영 리스크를 줄인다.

## 3) 내가 한 일(핵심 기여)

- **메시지 타입 도입에 따른 구조 확장**
    - 메시지 관련 데이터 구조를 확장하고, 변수 폼 UI 및 템플릿 연동 흐름을 구현
    - 임시저장/기존 오퍼 불러오기 기능에도 메시지 데이터 포함되도록 로직 확장
- **“일별 오퍼 예약 현황” 캘린더 UI**
    - 구글 캘린더와 유사한 구조로 학습 비용을 낮춘 UI 구성
    - hover 툴팁(핵심 정보) 제공 등 정보 접근성을 강화
    - 공식 가이드 전, 기획 기반 프로토타입을 빠르게 만들어 팀에 공유 → 초기안이 유지되는 방향에 기여
- **오퍼 관리/지급 현황 페이지 고도화**
    - 메시지 타입 검색 조건 추가, 결과 테이블 컬럼 추가 등 변경된 명세를 반영해 관리 편의 향상
- **API 이관(GET→POST) 시 영향도 분석으로 장애 예방**
    - 단순 교체가 아니라 코드베이스를 탐색해 동일 API 사용 지점을 사전에 찾아 공유하고 동시 적용하여 잠재 장애를 예방
- **정책 확인 모달(안전장치)**
    - 예외 정책(소규모 즉시 발송 등)에서 사고/중지 리스크가 있으므로 정책 안내 및 확인 모달을 추가해 오발송/운영 사고를 예방
- **QA/커뮤니케이션**
    - 스크럼/주간 회의/슬랙/기획서 댓글 등으로 불확실한 요구사항을 빠르게 확정하고, QA에서 발견된 엣지 케이스는 즉시 공유·티켓화하며 완성도를 끌어올림

## 4) 어려웠던 점과 해결

- API 스펙이 크게 바뀌면서 필터/요청/응답 구조 수정 범위가 커지고, RHF 기반 폼 구조에서 변경 지점을 추적하기 어려움
- 해결: 새 API 전용 “V2 흐름”을 분리 구현해 기존 코드를 보존하고 비교 가능하게 만들며 점진적으로 전환

# C4. 카카오페이 — 건틀렛 쿠폰 UI 변경(CSS vs SVG 대안 비교)

## 1) 배경/문제

- 쿠폰 홀(구멍) 형태의 새로운 디자인을 기존 반응형 리스트젬 UI에 맞춰 구현해야 했음

## 2) 대안 비교

- 1안(CSS): 배경색과 동일한 원을 겹쳐 쿠폰 홀 형태 구현 → 모서리 디테일이 시안과 달라 완성도 한계

- 최종: UI 최대 너비를 고정해 SVG 왜곡을 방지하는 방향으로 팀 논의 후 결정

## 3) 결과

- 디자인 가이드에 부합하는 쿠폰 UI 구현
- 부가 UX 개선: 체크 컬러 변경, 클릭 영역 확대 등 사용 편의성 개선

# C5. 별숲 — R3F 기반 3D 밤하늘 UX + 데이터 신선도 중심 설계 + API 처리 규칙/리팩터링

## 1) 배경/문제

- 3D 밤하늘 UX(탐색/시점 이동)가 핵심이며, CRUD/API 연동이 많은 서비스에서 안정적인 상태/에러 처리가 필수
- SPA에서 3D view ↔ 리스트 view 이동 및 뒤로가기 UX를 제공하면서도 데이터 freshness를 유지해야 함

## 2) 3D 배경 구현(“밤하늘” UX)

- 컨트롤 후보들을 검토하고, “고개를 돌려 하늘을 보는 느낌”을 위해 **OrbitControls**를 선택
- 밤하늘을 구 내부로 모델링하기 위해 **sphereGeometry**를 사용해 사용자가 구 내부에 위치하도록 구성
- 구가 부드럽게 보이도록 segment를 높이고, 위쪽 하늘만 필요해 pi/theta 조정으로 하단을 컷
- “바닥에서 하늘을 보는 시점”을 강화하기 위해 FBX 바닥 모델을 제작하고 useFBX로 로드/배치

## 3) 3D 환경 카메라 이동(더블클릭 시점 이동)

- onDoubleClick 이벤트에서 얻는 point를 활용해 Vector3 좌표를 확보
- 세 가지 대안을 비교:
    - OrbitControls target 활용: 별 생성/드래그에 의해 시점이 불규칙해지는 문제
    - Raycasting: 시점 변환이 어색하고 target 문제 해결 불가
    - 카메라 위치 이동(채택): target을 고정하고 카메라 위치를 이동해 문제를 깔끔하게 해결
- 결과적으로 더블클릭 좌표로 자연스럽게 밤하늘을 관찰할 수 있는 로직 완성

## 4) API 요청/응답 처리 규칙(상태 코드 기반)

- CRUD/데이터 fetch 등 다양한 요청에서 상태 코드 기반의 일관된 처리 흐름을 설계
- 특히 401(토큰 만료)에서 재발급 요청 후 새 토큰으로 원 요청을 재전송하는 흐름을 정의해 UX 단절 최소화

## 5) React-Query 공통화(리팩터링)

- 여러 모달/페이지에서 반복되는 res.status 분기 로직을 공통화하기 위해 패턴을 정리
- 성공/권한/만료(401 재발급) 흐름을 일관되게 구성해 유지보수성을 높임

## 6) SPA 페이지 이동 설계(History API vs Recoil)

- History API는 뒤로가기 UX는 좋지만 stale 데이터 리스크가 존재
- Recoil atom stack은 freshness 유지에 유리하나 브라우저 뒤로가기 UX 제약이 있음
- 별숲의 특성상 freshness 우선으로 Recoil 방식 채택(향후 혼합 개선 여지 인지)

## 7) 운영/협업 자산

- 스프린트/백로그/데일리 스크럼/회의록/개발일지/발표자료/개인 회고 등 기록 기반 운영
- Ground Rule 기반 팀 리듬(코어타임/오프라인 회의/루틴)을 합의해 커뮤니케이션 비용을 줄임
- 멘토링 일지(6주)로 질문→피드백→실행→회고를 반복하며 학습을 실행으로 연결

# C6. PONG! PONG! PONG! — 온라인/로컬 PONG 게임 서비스 (FE / GAME 기여 상세)

### 1) 프로젝트 개요

- 하나의 화면/키보드로 즐기는 **로컬 모드**와, 여러 PC로 진행하는 **온라인 모드**를 제공하는 PONG 게임 서비스.
- 온라인 모드는 **대기실 조회/생성/참가**, 커스텀 게임 설정(매개변수 조정) 및 파워업을 지원하며, 사용자 대시보드에서 친구/정보/통계를 확인할 수 있는 형태로 설계됨.
- FE 기술 스택은 **JavaScript, Bootstrap, Three.js**로 명시되어 있음.

### 2) 내가 맡은 역할(레포 README에 명시된 범위)

README의 “팀원 소개 및 역할 분담”에 따라, **joyoo(유종석)**의 담당 범위는 다음과 같다.

- 사용자 대시보드 **모달 UI 및 API 연결** *(통계 탭 제외, Bootstrap 활용)*
- **3D 게임 그래픽 및 애니메이션 구현** *(Three.js 활용)*
- **로컬 게임 웹소켓 연결 및 통신**
- **게임 계산 로직, 파워업**

### 3) 구현 상세(기여 영역을 “기술 단위”로 분해)

### 3-1. 사용자 대시보드 모달 UI + API 연결(통계 탭 제외)

- **목표**: 게임 플레이 외의 서비스 기능(내 정보/친구 등)을 UI로 제공하고, 모달 상호작용으로 빠르게 조회·수정이 가능한 경험을 제공.
- **접근**
    - Bootstrap 기반의 모달 UI로 구성해, 컴포넌트/레이아웃을 빠르게 안정화(모달 접근/닫기/포커스 흐름 등 UI 품질을 Bootstrap 규격에 맞게 가져감).
    - 모달에서 필요한 데이터는 API와 연결해 조회/반영 흐름을 구현(통계 탭은 제외).

### 3-2. Three.js 기반 3D 게임 그래픽 및 애니메이션

- **목표**: PONG 플레이를 3D로 시각화해 체감 품질을 높이고, 애니메이션을 통해 공/패들/환경 요소의 움직임을 자연스럽게 표현.
- **구현 범위(README 기준)**
    - Three.js를 활용한 **3D 그래픽 및 애니메이션 구현**을 담당.
- **기술적으로 신경 쓴 포인트(Three.js 특성 기반 정리)**
    - 실시간 게임에서는 렌더링 루프(프레임 단위 업데이트) 안에서 **오브젝트의 위치/속도/충돌 결과를 반영**해야 하므로, “게임 계산 로직”과 “렌더링”의 결합 지점을 명확히 관리하는 것이 중요함(아래 3-4와 연결).

### 3-3. 로컬 게임 WebSocket 연결 및 통신

- **목표**: 로컬 모드에서도 입력/상태 변화가 실시간으로 전달되어 게임이 끊김 없이 진행되도록 WebSocket 기반 연결/통신을 구현.
- **구현 범위(README 기준)**
    - “로컬 게임 웹소켓 연결 및 통신” 담당으로 명시.

### 3-4. 게임 계산 로직 및 파워업

- **목표**: PONG의 핵심 규칙(이동/충돌/득점 등)을 계산 로직으로 구현하고, 파워업을 통해 플레이 경험을 확장.
- **구현 범위(README 기준)**
    - “게임 계산 로직, 파워업” 구현 담당으로 명시.
- **설계 관점 정리(게임 로직 특성 기반)**
    - 실시간 게임 로직은 **(1) 상태 업데이트(물리/충돌/점수)**, **(2) 화면 반영(애니메이션/렌더링)**, **(3) 통신(WebSocket 이벤트)**이 동시에 맞물리기 때문에, 기능이 커질수록 모듈 경계를 의식하는 설계가 유지보수에 유리함(그래서 로직/렌더/통신의 책임을 분리하는 방향으로 관리하는 게 일반적으로 안전).

### 4) 결과

- 로컬/온라인 모드를 지원하는 PONG 서비스에서, 내 담당 범위(대시보드 모달+API, 3D 그래픽/애니메이션, 로컬 WebSocket, 게임 계산 로직/파워업)를 구현해 “서비스 기능 + 플레이 경험”을 함께 완성하는 데 기여.

# C7. Sleep Now! — 센서 기반 수면 직전 감지 + 알림 중재 앱 (React Native) 상세

### 1) 개발 배경

- 스마트폰 사용이 일상화되면서 “잠들기 직전 스마트폰 사용”이 증가했고, 밝은 화면 노출 등이 불면을 유발할 수 있다는 문제의식에서 출발.
- 기존의 제어 앱/OS 수면 기능은 있었지만, **“자기 직전 스마트폰 사용을 자제”**시키는 형태의 기능이 부재해 근본 해결이 어렵다고 보고 문제를 정의.

### 2) 목표

- 사용자가 누워서 잠들기 직전의 **자세 및 환경**이 인식되면, 스마트폰 사용을 자제하고 바로 수면하도록 유도하는 서비스를 개발.
- 핵심은 개인별 수면 시간대/자세가 달라질 수 있다는 전제에서 **유연하게 동작**하는 것.

### 3) 가정 및 시스템 개요(README 명시)

- 가정(요약): 수면 시간대/자세는 개인마다 다르며, 사용자는 수면 필요성을 인지하지만 사용 자제가 어렵다.
- 시스템 개요(요약):
    - 스마트폰 센서로 자세 데이터를 수집/분석
    - 사용자별 개인 맞춤 모델 수립
    - 백그라운드에서 끊김 없이 동작
    - 실시간 센서로 수면 직전 여부 판단
    - 수면을 유도하는 중재 기능 제공

### 4) 기술 구성(라이브러리/역할)

- **react-native-sensors**: Accelerometer / Gyroscope / Magnetometer 활용
- **react-native-ambient-light-sensor**: Light sensor(조도) 활용
- **nodejs-decision-tree**: 개인 맞춤 Decision Tree 모델 수립 목적
- **notifee**: Foreground Service 및 Push Notification 활용
- **react-native-async-storage**: 자동 로그인, 데이터 로딩 최적화 및 비용 감소 목적

### 5) 앱 동작 흐름(사용자 경험 + 데이터 흐름)

### 5-1. 로그인/메인 진입

- 로그인 여부에 따라 로그인/회원가입 화면 → 로그인된 화면으로 진입하는 흐름을 제공.

### 5-2. 센서 수집 상태 확인 및 모드 전환(핵심 화면)

- 메인 페이지에서 **실시간 센서 수집 여부**를 확인할 수 있고,
    - **데이터 수집 및 라벨링 토글**
    - **서비스 동작 모드(실시간 예측) 전환 버튼**을 제공.

### 5-3. 데이터 수집 모드 vs 예측 서비스 모드

- **실시간 예측 서비스 모드**: 백그라운드에서도 지속적으로 스마트폰 데이터를 수집하며, 실시간 분석/예측으로 “수면 직전 상황 여부”를 판단.
- **데이터 수집 모드**: 실시간 데이터를 **Firebase Firestore에 전송 후 저장**하여 학습/개인화에 활용 가능한 데이터 기반을 만든다.

### 5-4. 중재(Intervention): 알림으로 사용을 방해하고 수면 유도

- 수면 직전에 스마트폰을 사용하는 상황을 분석/예측하면, **소리·진동을 포함한 notification**을 출력해 사용을 방해하고 즉시 수면을 유도.

### 6) 모델 성능 관찰 및 한계(README 명시)

- 사용한 모델은 Decision Tree이며, 목표 과제가 크게 복잡하지 않아 **직접 수집한 데이터 기반 분석에서 accuracy 95% 이상**을 보였다고 기록.
- 다만 실제 서비스 동작 중 “자려는 상황이 아닌데 알림이 출력”되는 **오판단 케이스**가 발생했다고 명시.

### 7) 향후 계획(README/Issues 기반)

### 7-1. 개인화 모델 고도화

- parameter tuning(센서 값 가중치 등) 및 RandomForest, SVM 등 다른 알고리즘 도입/성능 테스트 계획을 제시.

### 7-2. 수면 권장 시스템 강화(중재 전략 강화)

- 단순 알림을 넘어 습관 변화를 유도하기 위해, 알림 주기 조정/스마트폰 제어(밝기·음량)/성취 요소/자동 메시지/커스텀 동기부여 메시지 등 다양한 개선 아이디어를 제시.

### 7-3. 포그라운드 서비스 운영 개선(이슈로 관리)

- 현재 “데이터 수집용 포그라운드”와 “알림 시작용 포그라운드”가 분리된 것을 **상시 포그라운드 서비스로 통일**하고, 수면 권장 알림은 **20시~4시**에만 동작하도록 하자는 개선안을 이슈로 관리.
- “GPS가 집일 때만 동작”시키는 아이디어를 이슈로 등록(설명은 제목 수준).
